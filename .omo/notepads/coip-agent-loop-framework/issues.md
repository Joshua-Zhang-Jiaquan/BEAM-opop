## tests/orchestrator/test_repro.py ‚Äî abstract-knob/param-name seam now LOCKED under a regression test
- **Confirms & formalizes the task-17 line-35 observation** (the throwaway smoke that hit `'Not a valid parameter name'`). `test_strict_replay_reproduces` is the permanent end-to-end coverage: real `Phase1Controller.bo(default_phase1_space())` + real `ScipKernel()` on `generate_knapsack(6, seed=0)`, then `replay_run(strict=True)` ‚Üí `REPRODUCED`, exit 0, matching incumbent objective + n_accepted.
- **Root cause pinned (NOT a test/replay bug; the controller‚Üîkernel param contract has a gap)**: the BO controller's decoded `phi.p` carries the ENCODER's abstract knob names ‚Äî `cut_aggressiveness` / `branching_scorefac` / `separating_maxrounds` (see `default_phase1_space` `ContinuousDictDim("p", ‚Ä¶)`) ‚Äî but `ScipKernel.apply_proposer_hooks` calls `model.setParam(key, value)` verbatim, and SCIP rejects those names with `KeyError('Not a valid parameter name')`. Verified directly: bare `Phi()` solve = optimal (primal 210); controller-`phi` solve = `KeyError`. So with REAL components EVERY solve is recorded+skipped ‚Üí `n_accepted==0`, `incumbent.json==null`. The `scip.py` docstring already defers "translating high-level phi.d/phi.h fields into concrete parameter sets ‚Ä¶ to the task-14 proposer", but `phi.p` knob names are emitted today and there is no translation/mapping layer; the curated `set_param` deltas in `proposer/params.py` DO use real SCIP paths, but they are merged on top of `phi.p` and the abstract `p` knobs poison the whole `apply_proposer_hooks` loop first.
- **Test is fix-tolerant**: it asserts original==replay (equality), NOT literal `n_accepted==0`/`incumbent is None`, so it stays green once an encoder-knob‚ÜíSCIP-param mapping (or a fail-soft `setParam`) lands and accepted counts go positive. Owner action for whoever owns `controller/encoder.py` + `solver/scip.py`: either map the abstract `p` keys to concrete SCIP params before `setParam`, or have `apply_proposer_hooks` skip/validate unknown keys instead of letting SCIP raise.

## Task 21 ‚Äî `branching/scorefactor` is NOT a real SCIP param (the task-17 "RESOLVED" claim was incomplete; CAUGHT by the end-to-end smoke)
- **Bug**: the task-17 fix swapped the encoder's ABSTRACT knob names for "real SCIP parameter paths", but one of the six curated keys ‚Äî `branching/scorefactor` ‚Äî is itself invalid. PySCIPOpt 6.2.1 / SCIP 10.0.2 `model.setParam("branching/scorefactor", ...)` raises `'Not a valid parameter name'`. The correct path is **`branching/scorefac`** (probed live: `branching/scorefac` OK, `branching/scorefactor` FAIL).
- **Impact (total, silent until now)**: `default_phase1_space()` derives `phi.p` keys from `CURATED_PARAMS`, so EVERY controller-proposed `phi` carried the bad key; `eff_phi` always merges `phi.p`, so EVERY `ScipKernel.solve` hit `apply_proposer_hooks` ‚Üí `setParam` ‚Üí raised ‚Üí recorded as `solve_error` and skipped. Result before the fix: `n_accepted == 0` for ALL opop rows, every `incumbent` null. The throwaway smokes in task-17/replay (issues line 35) and the fix-tolerant `test_strict_replay_reproduces` (asserts original==replay, NOT `n_accepted>0`) both stayed green and MASKED this ‚Äî only the task-21 end-to-end smoke, which asserts `n_accepted >= 1`, surfaced it. This is exactly the latent integration bug a Wave-3 smoke is meant to catch.
- **Fix (root cause)**: `src/opop/proposer/params.py` `CURATED_PARAMS` key `branching/scorefactor` ‚Üí `branching/scorefac`, and the 3 round-trip fixtures in `tests/controller/test_phase1.py` (lines 33/62/82) updated to match (they encode/decode against `default_phase1_space()`, whose `p` keys now derive from the corrected `CURATED_PARAMS`). After: `n_accepted == 10` per opop `(instance,seed)` (2 trials √ó 5 param deltas). **`params.py`/`test_phase1.py` were NOT in task-21's "Modified files" list, but `n_accepted >= 1` is a hard task requirement that is impossible while the controller emits an invalid SCIP key ‚Äî so this upstream fix was mandatory.** Owner of `proposer/params.py`: please also note the learnings.md line still says `branching/scorefactor` for the encoder section (now stale).
- **NOTE for whoever audits learnings.md**: the task-17 RESOLVED note + the "Phi encoder" learnings section both list `branching/scorefactor` as one of the "real SCIP parameter paths" ‚Äî that statement was wrong; the live-correct key is `branching/scorefac`.
